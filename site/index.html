Multi-tenancy that refuses to leak

A TypeScript-first tenancy toolkit for Node.js. Tenant identity follows your async execution scope — and any tenant-aware access without a valid context throws, instead of returning another tenant's data.

The failure mode

A forgotten WHERE clause should be a crash, not a data breach

The dangerous bug in multi-tenancy is the one that doesn't throw. A missing tenant filter returns data — silently — and you find out from a support ticket. TenancyJS makes the safe path the default: no valid tenant context, no data. 

// Tenant identity rides AsyncLocalStorage.
await tenancy.runWithTenant(tenant, async () => {
  // Every adapter scopes access to this tenant.
  const orders = await db.order.findMany();
});

// Outside a tenant scope? It fails closed.
await db.order.findMany();
// ✗ throws TenantContextError — never
//   a silent fallback to unscoped data
Three strategies, one contract

Pick your isolation. Keep your code.

The same tenant contract drives all three. Every supported cell is backed by a two-tenant adversarial isolation test on a real database.

01 — row-level

Single database

Shared tables keyed by tenant_id, enforced by forced Postgres RLS or query-scoping. The lightest footprint.

Knex · Lucid · Prisma · TypeORM · Sequelize · Mongoose
02 — schema-per-tenant

Schema per tenant

One Postgres schema per tenant via transaction-local search_path, with an optional per-tenant role for database-enforced isolation.

Knex · Lucid
03 — database-per-tenant

Database per tenant

A separate database per tenant, routed through a bounded, single-flight connection cache. Hard isolation, no noisy neighbours.

Knex · Lucid · Prisma

Operational CLI

Operate your tenants, honestly

The CLI reaches your live tenants through your store and hooks — it never invents ORM behaviour it hasn't tested. tenant check reads each adapter's own capability report and warns you about any untested combination, rather than pretending it's production-ready. Secrets are redacted from every command's output.

Honesty over faith

Guarantees, not vibes

Everything here is a property of the system, not of your vigilance.

Fail-closed, always

No valid tenant context means an error at the boundary — never a silent read of unscoped data.

Proven, not claimed

A capability is marked "supported" only after a real two-tenant adversarial test on a real database.

Bring your own store

Your registry, hardened at the boundary — a buggy store can't hand back the wrong tenant.

Cleanup always runs

Transaction-scoped context is torn down on every path, including errors — no leaked scope between requests.
