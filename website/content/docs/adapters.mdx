---
title: Adapters
description: Adapters enforce tenant isolation inside the ORM you already use. Here's what each one does and how to wire it.
---

An **adapter** is the piece that actually keeps tenants apart inside your ORM. You give it your ORM
client and tell it which models are tenant-scoped; it makes sure every query that runs inside a tenant
scope is filtered to that tenant — and that queries outside a scope fail closed.

You only need the adapter for the ORM you use. Check the [capability matrix](/docs/capability-matrix)
first to see which strategies each one supports.

## How adapters work

Every adapter follows the same shape:

1. You create it with your `manager` and your ORM client.
2. You declare the tenant-scoped models.
3. You use the adapter's client (or `run`) inside `manager.runWithTenant(...)`.

```ts
const tenancy = createXxxAdapter({ manager, client, tenantModels: { Post: {} } });

await manager.runWithTenant({ id: "acme" }, async () => {
  await tenancy.client.post.findMany(); // scoped to acme
});
```

## Prisma

`tenancyjs-adapter-prisma` — row-level (PostgreSQL + MySQL) and database-per-tenant.

```ts
import { createPrismaAdapter } from "tenancyjs-adapter-prisma";

const tenancy = createPrismaAdapter({
  manager,
  client: prisma,
  tenantModels: { Post: {}, Order: {} },
});
```

For database-per-tenant, use `createPrismaDatabaseTenancy` to route a dedicated client per tenant.

## Knex & Lucid

`tenancyjs-adapter-knex` and `tenancyjs-adapter-lucid` are the most complete — **all three strategies**
on PostgreSQL. Row-level and schema-per-tenant use forced Postgres **RLS** and a transaction-local
`search_path`; database-per-tenant leases a per-tenant connection from a bounded cache.

```ts
import { createKnexAdapter } from "tenancyjs-adapter-knex";

const tenancy = createKnexAdapter({
  manager,
  knex,
  strategy: "rowLevel", // or "schemaPerTenant" | "databasePerTenant"
  tenantTables: ["posts", "orders"],
});
```

## TypeORM & Sequelize

`tenancyjs-adapter-typeorm` and `tenancyjs-adapter-sequelize` provide **row-level** isolation on
PostgreSQL today (forced RLS + a query facade). Schema- and database-per-tenant are on the roadmap.

## Mongoose

`tenancyjs-adapter-mongoose` scopes MongoDB access per tenant.

<Callout type="warn">
  MongoDB has no row-level security, so Mongoose isolation is **facade-only** — it holds as long as you
  go through the adapter. Using the native model, collection, or connection bypasses it entirely. Never
  expose those outside the adapter.
</Callout>

## Defense in depth

SQL adapters use two layers: a whitelist facade **and** forced database RLS, so a bug in one is caught
by the other. Mongoose has only the facade — which is why the docs (and `tenant check`) call it out
explicitly.
