title: Resolving tenants
description: Turn an incoming request into a tenant - from subdomains, headers, paths, or your store.
---

Every [integration](/docs/integrations) takes a **resolver**: a function that turns a request into a
tenant. This guide covers the common patterns and how to fail closed correctly.

## The shape

A resolver returns the tenant (or enough to identify it). The simplest form reads an identifier
straight off the request:

```ts
resolver: (req) => ({ id: req.subdomains.at(-1) ?? "" });
```

If it can't produce a valid tenant, the integration responds with the right status and your handler
never runs.

  </Tab>

## Looking up and validating

For anything beyond reading a string, resolve against your store - and let unknown or suspended tenants
fail closed:

```ts
resolver: async (req) => {
  const slug = req.subdomains.at(-1);
  if (!slug) return null; // 400 - no identifier
  const tenant = await store.find(slug);
  if (!tenant || tenant.status === "suspended") return null; // 404
  return tenant;
};
```

## Consistent failure semantics

`tenancyjs-identifiers` gives you composable resolvers and a single, consistent mapping from resolution
outcome to HTTP status. Compose the resolvers you need and give the chain a store to look the identifier
up in:

```ts
import {
  HeaderTenantResolver,
  SubdomainTenantResolver,
  TenantResolutionChain,
} from "tenancyjs-identifiers";

const chain = new TenantResolutionChain<Tenant>({
  resolvers: [
    new HeaderTenantResolver({ headerName: "x-tenant-id" }),
    new SubdomainTenantResolver({ centralDomain: "app.example.com" }),
  ],
  store: {
    // Look the identifier up in YOUR store and return the matches.
    async find(identifier) {
      const tenant = await lookupTenant(identifier.value);
      return tenant ? [{ tenant, status: "active" }] : [];
    },
  },
});

const outcome = await chain.resolve({ host: req.host, headers: req.headers });
if (outcome.status === "resolved") {
  // outcome.tenant — hand it to manager.runWithTenant(...)
}
```

Outcomes map to a status consistently — **400** for `no-identifier`/`invalid`, **404** for
`not-found`/`suspended` (deliberately indistinguishable), **500** for `ambiguous` (more than one match) —
so every framework behaves the same way. The Express/Next/Adonis integrations apply this for you; reach
for the chain directly when you resolve tenants by hand.

<Callout type="warn">
  This resolution store — `find(identifier) → { tenant, status }[]` — is **not** the same interface as the
  [registry store](/docs/getting-started/configuration#bring-your-own-tenant-store) the CLI uses (whose
  `find(id)` returns one tenant or `null`). Same idea, different shapes — don't reuse one object for both.
  And a resolver must never guess a tenant or fall back to a default; no match is the safe, fail-closed
  outcome.
</Callout>
