## Requirements

- **Node.js 24+** — the toolkit and CLI use modern Node features, and the CLI loads your TypeScript
  config via Node's native type-stripping (no transpiler needed).
- TypeScript is recommended but not required.

## Two independent choices: framework × ORM

TenancyJS composes from two axes that **don't need to know about each other**:

1. An **integration** for your framework — Express, Next.js, NestJS, or AdonisJS.
2. An **adapter** for your ORM — Prisma, Knex, Lucid, TypeORM, Sequelize, or Mongoose.

Any integration works with any adapter. **Express + Sequelize**, **Next.js + Knex**,
**NestJS + Mongoose** — all valid; both pieces just talk to the same core `TenancyManager`. So there's
no fixed "stack": you pick your framework's integration and your ORM's adapter, plus `tenancyjs-core`.

<Callout type="info">
  Wondering if your combination is supported? The framework and ORM are orthogonal — every integration ×
  every adapter works. What *does* vary is which **isolation strategy** each adapter supports (row-level
  vs schema/database-per-tenant) — that's the [capability matrix](/docs/concepts/capability-matrix).
</Callout>

## Fastest path: scaffold with the CLI

`tenancy init` writes a working `tenancy.config.ts`, tenant registration, and request middleware for
you — no boilerplate.

```bash
npx tenancy init
```

Today it ships ready-made templates for these three popular combinations:

| Framework | ORM | `tenancy init` |
| --- | --- | :---: |
| Express | Prisma | ✅ scaffolded |
| AdonisJS | Lucid | ✅ scaffolded |
| Next.js | Prisma | ✅ scaffolded |

Add the packages it wired (for Express + Prisma):

```bash
npm install tenancyjs-core tenancyjs-adapter-prisma tenancyjs-integration-express
```

Then jump to the [Quickstart](/docs/getting-started/quickstart).

<Callout type="info">
  Any other combination — **Express + Sequelize**, **Next.js + Knex**, **NestJS + anything** — isn't
  scaffolded *yet*, but it's a few lines of manual wiring. Follow [manual setup](#manual-setup) below.
</Callout>

### The CLI does more than scaffold

`init` is just the start. The same CLI lists, creates, migrates, provisions, and runs scripts against
your live tenants. See the full **[CLI reference →](/docs/cli)**.

## Manual setup

Works for **any** framework × ORM. Install three things: the core, your framework's **integration**, and
your ORM's **adapter**. For example, Express with Sequelize:

<Tabs items={["npm", "pnpm", "yarn", "bun"]}>
  <Tab value="npm">```bash npm install tenancyjs-core tenancyjs-integration-express tenancyjs-adapter-sequelize ```</Tab>
  <Tab value="pnpm">```bash pnpm add tenancyjs-core tenancyjs-integration-express tenancyjs-adapter-sequelize ```</Tab>
  <Tab value="yarn">```bash yarn add tenancyjs-core tenancyjs-integration-express tenancyjs-adapter-sequelize ```</Tab>
  <Tab value="bun">```bash bun add tenancyjs-core tenancyjs-integration-express tenancyjs-adapter-sequelize ```</Tab>
</Tabs>

Then follow the two guides for your choices — the [integration guide](/docs/integrations) for the
framework half and the [adapter guide](/docs/adapters) for the ORM half. That's the whole wiring.

| Package | Role |
| --- | --- |
| `tenancyjs-core` | Framework-neutral tenant context, lifecycle, and the runtime contract |
| `tenancyjs-identifiers` | Resolve a tenant from a request (subdomain, header, path…) |
| `tenancyjs-adapter-prisma` · `-knex` · `-lucid` · `-typeorm` · `-sequelize` · `-mongoose` | Enforce isolation inside your ORM ([guides](/docs/adapters)) |
| `tenancyjs-integration-express` · `-next` · `-adonis` · `-nest` | Bind context to the request lifecycle ([guides](/docs/integrations)) |
| `tenancyjs-cli` | The operational CLI (`tenancy …`) |
| `tenancyjs-testing` | Fixtures and conformance helpers for your own tests |

<Callout type="warn">
  During the beta the packages are published under the `beta` dist-tag. A plain `npm install
  tenancyjs-core` resolves the latest published beta; pin an exact version for reproducibility.
</Callout>
