This page wires fail-closed multi-tenancy into an AdonisJS 7 + Lucid 22 app — **new or existing**. Two
ways to do it: hand the **integration-agent prompt** to your AI assistant, or follow the **manual
walkthrough**. The agent assesses your app first — `npx tenancyjs-cli init` can scaffold this stack from
scratch if you're starting greenfield — integrates with your existing models and auth, and finishes with
an isolated leak test, because with fail-closed isolation you verify, you don't trust.

This is **row-level** on PostgreSQL. Lucid row-level is a **callback facade backed by forced RLS**: the
adapter scopes model operations through Lucid's hooks, and Postgres row-level security is the backstop
that refuses cross-tenant rows even if a query slips the facade. That database enforcement is why this
stack forces RLS - the runtime role can't read another tenant's rows no matter what SQL reaches it.

<Callout type="info">
  Running **MySQL**? This page is PostgreSQL (row-level + forced RLS). MySQL has no schema-per-tenant
  and its row-level mode is experimental and facade-only - use
  [database-per-tenant on MySQL](/docs/stacks/mysql-database-per-tenant) instead.
</Callout>

<Callout type="info">
  **All three strategies for AdonisJS + Lucid are on this page.** Row-level is first (below); scroll down
  for the full integration-agent prompts for
  [schema-per-tenant →](#integration-agent-prompt--schema-per-tenant) and
  [database-per-tenant →](#integration-agent-prompt--database-per-tenant).
</Callout>

````text
## Step 1 - Scaffold or wire manually
`npx tenancyjs-cli init` scaffolds this exact stack end to end. Offer to run it, then adapt the result to
this repo. If the user prefers explicit wiring (or `init` doesn't fit), do Steps 2–8 by hand.

## Step 2 - Install
Run:
```sh
npm install tenancyjs-core tenancyjs-adapter-lucid tenancyjs-integration-adonis tenancyjs-identifiers
```
Verify all four appear in package.json dependencies. `tenancyjs-identifiers` MUST be a direct
dependency - AdonisJS only resolves it transitively otherwise, and config import fails.

## Step 3 - config/tenancy.ts (the single source of truth)
Create `config/tenancy.ts`. It must:
- Construct ONE `TenancyManager<Tenant>`.
- Build the Lucid adapter with `createLucidTenancy({ manager, database: db, tenantModels: [...] })`
  (`db` from `@adonisjs/lucid/services/db`). Give each EXISTING tenant-owned model an explicit entry:
  `{ model: Post, tenantAttribute: "tenantId", tenantColumn: "tenant_id", policyName: "posts_tenant_isolation" }`.
  Only add a new example model if the app has no domain models yet. If a model is missing its
  `tenant_id` column, add it via migration and backfill existing rows before enforcing non-null (see
  Step 9 - Roll out).
- Build a `TenantResolutionChain` with `new HeaderTenantResolver()` and a store whose `find(identifier)`
  returns `[{ tenant: { id }, status }]` (or `[]`).
- `x-tenant-id` is client-forgeable, so the chain REQUIRES a membership check: pass an
  `authorize: ({ tenant, principal }) => principal.teamIds.includes(tenant.id)` hook (it throws to
  construct without one). Do not use `trustResolution` with a raw header resolver.
- Export `defineAdonisTenancyConfig({ manager, resolver, principal, tenancy })` as default, passing
  `principal: (ctx) => ctx.auth.user` (register the tenant middleware AFTER auth) and `tenancy` as a
  lazy factory `() => createLucidTenancy(...)` (config loads before providers boot).

## Step 4 - Forced-RLS migration
Create a migration that installs RLS as the DB OWNER and runs the app as a NON-owner runtime role:
- The runtime role must NOT own the tables, must NOT have BYPASSRLS, and must NOT be a superuser
  (or FORCE RLS is silently skipped for it).
- On each tenant table: `ENABLE ROW LEVEL SECURITY` and `FORCE ROW LEVEL SECURITY`.
- Create policy `posts_tenant_isolation` whose USING and WITH CHECK both read the request GUCs:
  `tenant_id = current_setting('tenancyjs.tenant_id', true)` OR
  `current_setting('tenancyjs.is_central', true) = 'true'`.
Verify the policy exists (`\d+ posts`) and the runtime role is non-owner.

## Step 5 - Register provider + middleware
- In `adonisrc.ts`, add `() => import('tenancyjs-integration-adonis/provider')` to `providers`.
- Add `app/middleware/tenant_middleware.ts` re-exporting `TenancyMiddleware` as default, register it as
  a NAMED middleware `tenant` in `start/kernel.ts`.
- In `start/routes.ts`, apply `.use(middleware.tenant())` to tenant route GROUPS only. Central routes
  omit it - never apply it globally.

## Step 6 - Validate at startup
Ensure `await tenancy.validate()` runs before the first `tenancy.run(...)` and THROWS if invalid (the
web provider validates automatically; Ace commands and scripts must call it themselves).

## Step 7 - Central (cross-tenant) work
For admin/cross-tenant handlers, do NOT leave them unscoped. Open an explicit central scope through
application-owned code (`manager.runInCentralContext(...)`); unscoped tenant access must fail closed.

## Step 8 - Verify isolation (an isolated test - it does NOT touch app data)
Create a two-tenant leak TEST (`node ace test`) that runs against a TEST database (or a rolled-back
transaction) - never seed tenants into real app data. Write a row under tenant A and tenant B with the
SAME primary key; assert querying as A never returns B's row; and assert that raw SQL run as the runtime
role with NO GUC set returns 0 rows (RLS backstop). In tests, call `config.tenancy.validate()` yourself -
the /testing helper does not auto-validate. Follow https://tenancyjs.pages.dev/docs/guides/testing-isolation.

## Step 9 - Roll out + verify
- `node ace build` - report errors.
- Boot the app (`node ace serve`) and confirm it serves.
- Run `node ace test` and report the result.
- Existing app: backfill `tenant_id` on any table that already had rows (set it for every row, then
  enforce non-null and FORCE RLS), then roll out incrementally - scope routes behind the middleware, keep
  admin/cross-tenant work in `runInCentralContext`, and treat any thrown unscoped-access error as a route
  you still need to scope.

Report a summary of every file created/changed and any issue found.
````

<Callout type="warn">
  AI wiring is a starting point, not proof. The only thing that demonstrates isolation is the two-tenant
  colliding-id test in Step 8 - run it, including the raw-as-runtime-role check, against a test database
  before you trust it.
</Callout>

## Manual walkthrough

Prefer to wire it yourself? The same steps, with the code.

<Steps>

<Step>
### Install

```bash
npm install tenancyjs-core tenancyjs-adapter-lucid tenancyjs-integration-adonis tenancyjs-identifiers
```

`@adonisjs/lucid`, `luxon`, and `pg` are AdonisJS peers - make sure `@adonisjs/lucid` and `pg` are
present. Keep `tenancyjs-identifiers` a **direct** dependency; Adonis only resolves it transitively

</Step>

<Step>
### One `config/tenancy.ts`

The manager, the Lucid adapter, and the resolver all live here - one source of truth. The adapter is a
callback facade; the AdonisJS config wraps it as a lazy factory.

```ts title="config/tenancy.ts"
import { TenancyManager } from "tenancyjs-core";
import { createLucidTenancy } from "tenancyjs-adapter-lucid";
import { HeaderTenantResolver, TenantResolutionChain } from "tenancyjs-identifiers";
import { defineAdonisTenancyConfig } from "tenancyjs-integration-adonis";
import db from "@adonisjs/lucid/services/db";
import Post from "#models/post";

export interface Tenant {
  readonly id: string;
}

const manager = new TenancyManager<Tenant>();

const tenancy = createLucidTenancy({
  manager,
  database: db,
  tenantModels: [
    {
      model: Post,
      tenantAttribute: "tenantId",
      tenantColumn: "tenant_id",
      policyName: "posts_tenant_isolation",
    },
  ],
});

const resolver = new TenantResolutionChain<Tenant>({
  resolvers: [new HeaderTenantResolver({ headerName: "x-tenant-id" })],
  store: {
    async find(identifier) {
      const row = await lookupTenant(identifier.value); // your DB lookup
      return row ? [{ tenant: { id: row.id }, status: row.status }] : [];
    },
  },
  // Required: does THIS authenticated user belong to the resolved tenant? (x-tenant-id is client-forgeable)
  authorize: ({ tenant, principal }) =>
    (principal as { teamIds: string[] }).teamIds.includes(tenant.id),
});

export default defineAdonisTenancyConfig<Tenant>({
  manager,
  resolver,
  principal: (ctx) => ctx.auth.user, // the authenticated user; run tenant middleware after auth
  tenancy: () => tenancy, // lazy: config loads before providers boot
});
```
</Step>

<Step>
### Force RLS in a migration

The facade is one guard; forced RLS is the database backstop. Install the tables and policy as the DB
**owner**, and run the app as a **non-owner** runtime role (no `BYPASSRLS`, not a superuser - otherwise
`FORCE` is skipped for it).

```ts title="database/migrations/xxxx_force_rls_posts.ts"
import { BaseSchema } from "@adonisjs/lucid/schema";

export default class extends BaseSchema {
  async up() {
    // run as the OWNER role that owns the table
    this.schema.raw(`ALTER TABLE posts ENABLE ROW LEVEL SECURITY`);
    this.schema.raw(`ALTER TABLE posts FORCE ROW LEVEL SECURITY`);
    this.schema.raw(`
      CREATE POLICY posts_tenant_isolation ON posts
      USING (
        tenant_id = current_setting('tenancyjs.tenant_id', true)
        OR current_setting('tenancyjs.is_central', true) = 'true'
      )
      WITH CHECK (
        tenant_id = current_setting('tenancyjs.tenant_id', true)
        OR current_setting('tenancyjs.is_central', true) = 'true'
      )
    `);
  }

  async down() {
    this.schema.raw(`DROP POLICY IF EXISTS posts_tenant_isolation ON posts`);
  }
}
```

The adapter sets `tenancyjs.tenant_id` (and `tenancyjs.is_central` in central scope) per transaction;
the policy reads them. No GUC ⇒ no rows.
</Step>

<Step>
### Register the provider and middleware

```ts title="adonisrc.ts"
providers: [
  // ...existing providers
  () => import("tenancyjs-integration-adonis/provider"),
],
```

```ts title="app/middleware/tenant_middleware.ts"
export { TenancyMiddleware as default } from "tenancyjs-integration-adonis";
```

```ts title="start/kernel.ts"
export const middleware = router.named({
  // ...existing named middleware
  tenant: () => import("#middleware/tenant_middleware"),
});
```

Apply it to tenant route **groups** only - central routes stay unmarked:

```ts title="start/routes.ts"
router
  .group(() => {
    // your tenant-scoped routes
  })
  .use(middleware.tenant());
```
</Step>

<Step>
### Run app work inside both contexts

The middleware opens the request's tenant scope; run the manager scope and the Lucid scope together, and
your models read isolated automatically - no `where tenantId`:

```ts title="app/controllers/posts_controller.ts"
const posts = await manager.runWithTenant(tenant, () =>
  tenancy.run(async () => {
    const posts = await Post.query().preload("comments");
    return posts;
  }),
);
```

Validate the adapter at startup and fail closed if it can't prove isolation:

```ts
const { valid } = await tenancy.validate();
if (!valid) throw new Error("Tenancy validation failed");
```

Cross-tenant admin work runs through an explicit central scope in application-owned code - never opened
by a request.
</Step>

<Step>
### Prove isolation

Write the [two-tenant colliding-id test](/docs/guides/testing-isolation) with `node ace test`: same
primary key under A and B, assert A never reads B's row, **and** assert a raw query run as the runtime
role with no GUC returns 0 rows (the RLS backstop). In tests, call `config.tenancy.validate()` yourself -
the `/testing` helper does not auto-validate.
</Step>

</Steps>

<Callout type="info">
  In Ace commands, scripts, and tests, call `tenancy.validate()` yourself - only the web provider
  validates automatically. A resolution failure fails closed, never central.
</Callout>
