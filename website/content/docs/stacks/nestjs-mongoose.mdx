`tenancyjs-cli init` doesn't scaffold NestJS yet, so this stack is a manual wire-up - a few files, for a
**new or existing** NestJS + Mongoose app. This page gives you two ways to do it: hand the
**integration-agent prompt** to your AI assistant, or follow the **manual walkthrough**. The agent
assesses your app first, integrates with your existing models and auth, and finishes with an isolated
leak test - because with fail-closed isolation you verify, you don't trust.

This is **row-level** on MongoDB. Mongoose is _facade-enforced_: MongoDB has **no row-level security**,
so there is no database backstop - the adapter's query facade is the entire boundary. The one rule that
matters most: never touch the native connection, model, or collection inside a scope - only the
protected facade the adapter hands you.

<Callout type="info">
  TenancyJS ships **dual ESM + CommonJS** (0.1.0+), so a default CommonJS NestJS project works with no
  conversion — `require` or import it as usual.
</Callout>

<Callout type="info">
  This page uses **row-level**. Mongoose also supports
  [database-per-tenant](/docs/stacks/database-per-tenant) (MongoDB has no schema-per-tenant) — switch
  the adapter’s `strategy` and add the [provisioning recipe](/docs/guides/provisioning/mongoose).
</Callout>

## Integration-agent prompt

Paste this into an assistant that can edit your repo and run commands (Claude Code, Cursor, …). It
assesses your app first, integrates with what already exists, verifies each step, and finishes with an
isolated leak test. It's pinned to the real API, so it won't invent factory names.

````text
# TenancyJS Integration Agent - NestJS 11 + Mongoose (row-level, MongoDB)

You are an integration agent. Wire fail-closed multi-tenancy into THIS project with TenancyJS,
integrating with whatever already exists — do NOT assume a greenfield app. Execute steps in order;
after each, verify it applied before continuing. If a step does not apply to this repo, say so and
adapt. Do NOT invent or assume API names, versions, or enum values, and never assert them from memory (your training data may be stale — library versions change). Verify against the real thing FIRST, then claim: `npm view <pkg> version` for a version, and the installed package's own types (in node_modules) or the linked docs for exact signatures and literal option values (e.g. the strategy is the string `"rowLevel"`).

Source of truth (fetch if a detail is missing, do not guess):
- https://tenancyjs.pages.dev/docs/adapters/mongoose
- https://tenancyjs.pages.dev/docs/integrations/nestjs
- https://tenancyjs.pages.dev/docs/concepts/limitations
- https://tenancyjs.pages.dev/docs/guides/testing-isolation
- https://tenancyjs.pages.dev/docs/guides/resolving-tenants

## Step 0 - Assess the app (change nothing yet)
- Report the Node version (abort if < 24), the `@nestjs/core` (v11) and `mongoose` versions, and confirm
  MongoDB is running as a REPLICA SET - Mongoose isolation runs inside managed transactions, which
  require a replica set; a standalone `mongod` will fail. If this is not a NestJS + Mongoose app, stop.
- Inventory the Mongoose models: list every model and mark each as tenant-owned or global/central.
- Find how the current user is determined per request (NestJS guard, `req.user`) — the `authorize` hook
  will need it.
- Detect any EXISTING tenant concept (a `tenantId`/`orgId` field, a workspaces/orgs collection). If one
  exists, REUSE it — do not add a parallel one.
- Summarize what you found + the integration plan, and confirm the ISOLATION STRATEGY with the user:
  row-level (this page) vs database-per-tenant (MongoDB has no schema-per-tenant). If row-level does not
  fit, point them to https://tenancyjs.pages.dev/docs/strategies/choosing and stop.

## Step 1 - Install
Run:
```sh
npm install tenancyjs-core tenancyjs-adapter-mongoose tenancyjs-integration-nest tenancyjs-identifiers mongoose
```
Verify all five appear in package.json dependencies.

## Step 2 - Models: classify what exists, add only what's missing
- Ensure a `Tenant` model (or store) exists holding `id` and `status`. If the app already has a
  tenants/workspaces/orgs collection, use THAT — do not add a parallel one.
- For every EXISTING tenant-owned Mongoose model, add a `tenantId` field. Only create a new example
  model (e.g. `PostModel`) if the app has no domain models yet.
- If you add `tenantId` to a collection that already has documents, backfill it before enforcing it as
  required (see Step 9 - Roll out).
Verify the models compile.

## Step 3 - tenancy.ts (the single source of truth)
Create `src/tenancy.ts`. It must:
- Construct ONE `TenancyManager<Tenant>` and export it.
- Open a Mongoose `connection` to the replica set:
  `await createConnection(process.env.MONGODB_URL).asPromise()`.
- Create the tenancy facade:
  `createMongooseTenancy({ manager, connection, tenantModels: [{ model: PostModel, tenantField: "tenantId" }] })`.
  Register EVERY tenant-owned model here. This `tenancy` is the ONLY handle the app may query through.
- Build a `TenantResolutionChain` with `new HeaderTenantResolver()` and a store whose `find(identifier)`
  looks the tenant up and returns `[{ tenant: { id }, status }]` (or `[]`). Include an
  `authorize({ tenant, principal }) => boolean` membership check that fails closed (a resolved tenant is
  not an authorized one); the chain refuses to construct without it.
- Call `await tenancy.validate()` at boot and REVIEW its output - it reports the enforcement tier and
  warns that Mongoose row-level has no database backstop.
- Export `manager`, `resolver`, and `tenancy`. Do NOT export the native `connection`, model, or
  collection for app use.

## Step 4 - Register TenancyModule
In `app.module.ts`, add `TenancyModule.forRoot({ manager, resolver })` to imports (import both from
`./tenancy`). Also pass `principal: (req) => req.user` so the guard hands the authenticated user to the
chain's `authorize` check (register it after your auth). Do not register a second guard/interceptor.

## Step 5 - Scope the routes
For every controller handler that touches tenant data:
- Add `@TenantRoute()` (from `tenancyjs-integration-nest`).
- Query through the facade: `tenancy.run((client) => client.model(PostModel).find({ status: "open" }))`.
  REMOVE any manual `tenantId` filters - the facade injects the predicate on reads and the field on
  writes. Leave non-tenant/public routes unmarked.

## Step 6 - Central (cross-tenant) work
For admin/cross-tenant handlers, do NOT leave them unscoped. Wrap the work in
`manager.runInCentralContext(() => tenancy.run(...))` explicitly. Unscoped tenant access must throw.

## Step 7 - Lock the boundary
- Never expose or use the native `connection`, `.model()` off the raw connection, or the native
  collection inside a scope. On MongoDB the facade IS the boundary - there is no RLS to catch a bypass.
- The facade rejects `$`-operator and dotted keys. If you need aggregation/populate, that's a
  database-per-tenant concern (see the adapter doc), not row-level.

## Step 8 - Verify isolation (an isolated test - it does NOT touch app data)
Create a two-tenant test (Vitest or Jest) that runs against a TEST database or a rolled-back transaction
- never seed tenants into real app data. It creates tenant A and tenant B, writes a document under EACH
with the SAME `_id`, asserts querying as A never returns B's document, and asserts that a tenant query
with no active scope THROWS. Because MongoDB has no RLS, this test is the ONLY proof the facade holds -
follow https://tenancyjs.pages.dev/docs/guides/testing-isolation.

## Step 9 - Roll out + verify
- `npx nest build` (or `npm run build`) - report errors.
- Boot the app (`npm run start`) and confirm it serves and `tenancy.validate()` passed.
- Run the two-tenant test and report the result.
- Existing app: backfill `tenantId` on any collection that already had documents (set it for every
  document, then enforce it as required), then roll out incrementally - scope routes behind
  `@TenantRoute()`, keep admin/cross-tenant paths in `runInCentralContext`, and treat any thrown
  unscoped-access error as a route you still need to scope.

Report a summary of every file created/changed and any issue found.
````

<Callout type="warn">
  AI wiring is a starting point, not proof. On MongoDB there is no database backstop, so the two-tenant
  colliding-`_id` test in Step 8 is the *only* thing that demonstrates isolation - run it against a test
  database before you trust the result.
</Callout>

## Manual walkthrough

Prefer to wire it yourself? The same steps, with the code.

<Steps>

<Step>
### Install

```bash
npm install tenancyjs-core tenancyjs-adapter-mongoose tenancyjs-integration-nest tenancyjs-identifiers mongoose
```
</Step>

<Step>
### One `tenancy.ts`

The manager, the Mongoose facade, and the resolver all live here - one source of truth. The connection
must reach a **replica set** (isolation runs in managed transactions).

```ts title="src/tenancy.ts"
import { TenancyManager, type TenantRecord } from "tenancyjs-core";
import { createMongooseTenancy } from "tenancyjs-adapter-mongoose";
import { HeaderTenantResolver, TenantResolutionChain } from "tenancyjs-identifiers";
import { createConnection } from "mongoose";
import { PostModel } from "./models";

interface Tenant extends TenantRecord {
  readonly id: string;
}

export const manager = new TenancyManager<Tenant>();

// Keep the native connection private to this module - it is the bypass.
const connection = await createConnection(process.env.MONGODB_URL!).asPromise();

// The ONLY handle the app may query through. Register every tenant-owned model.
export const tenancy = createMongooseTenancy({
  manager,
  connection,
  tenantModels: [{ model: PostModel, tenantField: "tenantId" }],
});

await tenancy.validate(); // review the enforcement-tier warning before serving

export const resolver = new TenantResolutionChain({
  resolvers: [new HeaderTenantResolver()], // X-Tenant-Id
  store: {
    async find(identifier) {
      const tenant = await TenantModel.findById(identifier.value);
      return tenant ? [{ tenant: { id: tenant.id }, status: tenant.status }] : [];
    },
  },
  // Required: does THIS authenticated user belong to the resolved tenant?
  authorize: ({ tenant, principal }) =>
    (principal as { teamIds: string[] }).teamIds.includes(tenant.id),
});
```
</Step>

<Step>
### Register the module

```ts title="src/app.module.ts"
import { Module } from "@nestjs/common";
import { TenancyModule } from "tenancyjs-integration-nest";
import { manager, resolver } from "./tenancy";

@Module({
  imports: [
    TenancyModule.forRoot({
      manager,
      resolver,
      principal: (req) => (req as { user?: unknown }).user, // the guard runs after auth
    }),
  ],
})
export class AppModule {}
```
</Step>

<Step>
### Scope your routes

Mark tenant routes; the interceptor opens the scope and the facade reads it - no manual `tenantId`
filter. Query only through `tenancy.run(...)`, never the native connection.

```ts title="src/posts.controller.ts"
import { Controller, Get } from "@nestjs/common";
import { TenantRoute } from "tenancyjs-integration-nest";
import { tenancy } from "./tenancy";
import { PostModel } from "./models";

@Controller("posts")
export class PostsController {
  @Get()
  @TenantRoute()
  list() {
    return tenancy.run((client) =>
      client.model(PostModel).find({ status: "open" }),
    ); // scoped to the request's tenant
  }
}
```
</Step>

<Step>
### Prove isolation

Write the [two-tenant colliding-`_id` test](/docs/guides/testing-isolation): same `_id` under A and B,
assert A never reads B's document, and assert an unscoped tenant query throws. On MongoDB the facade is
the whole boundary - with no RLS behind it, this test is your proof.
</Step>

</Steps>

<Callout type="info">
  Cross-tenant admin work runs in an explicit central scope -
  `manager.runInCentralContext(() => tenancy.run(...))`. There is no request-controlled central mode; a
  resolution failure fails closed, never central.
</Callout>
